# Design: Persistent, portable backing store (git + NAS)

Status: **approved direction (2026-06-21)** — this doc captures the decisions from
the design discussion and the build plan. Some pieces are marked **SPIKE** where a
mechanism still needs to be proven.

## 1. Problem

paddock's data lives on one disposable LXC's local disk with **no backup**:

- **Project content** — `/var/lib/paddock/projects/<slug>/` (project.yaml, OVERVIEW.md, CHANGELOG.md, authored notes/HTML). Not version-controlled.
- **Chat transcripts** — Claude Code JSONL at `~/.claude/projects/<encoded-cwd>/*.jsonl`, **keyed by the absolute working-directory path** (path-coupled → not portable).
- **Attribution** — `.herdctl/jobs/*.yaml` maps `session_id → agent`, i.e. which chat belongs to which project. Lives in herdctl's state dir, not with the project.

Three consequences we want to fix: (a) lose the LXC → lose everything; (b) projects aren't portable (clone/move breaks chats); (c) authored content piles up "saved to disk but never committed" with no visibility — the Zellij-tab problem.

## 2. Goals

1. **Durable** — data survives the LXC; the LXC is disposable (rebuild from backup).
2. **Portable / self-contained** — a project directory carries everything (content + its chats + their attribution); cloning or moving it "just works."
3. **Git as a first-class capability** — when the store is a git repo, surface uncommitted changes per project, and offer commit + push from the UI (the headline workflow feature, not just backup).
4. **Not git-only** — the base store is a plain directory; git is an optional layer; the remote is pluggable.

## 3. Decisions (locked)

- **The store is a filesystem directory.** Keeper/sweeper agents are Claude Code processes that need a real working dir, so the live representation is always a directory. Git / NAS / remotes are durability+sync *layers* on top of that dir.
- **Working-tree location = Option B.** The working tree stays on the LXC's **local disk** (fast, safe git). The **NAS hosts a bare repo** that is `origin`; paddock pushes to it continuously. Rebuild the LXC = `git clone` from the NAS. We never run live git over a network filesystem.
- **Transcripts relocate into each project** at `<project>/.chats/` so the project dir is self-contained and rides the same backup. (Mechanism: SPIKE in §5.)
- **Claude auth stays in env** (systemd `EnvironmentFile`), never a NAS mount.
- **In-app GitHub auth via OAuth device flow** (no public callback needed on a LAN box). **SSH deploy key** is the universal baseline for any git host.
- **Commit model:** auto-commit generated/derived files (OVERVIEW.md, CHANGELOG.md, transcripts) for durability; surface authored-file changes as "uncommitted" with a one-click **Commit + Push** in the UI for intentional checkpoints. (Default; tunable — see §8.)

## 4. Repo / data layout (self-contained projects)

```
<projects-repo>/                     # the working tree (local on LXC; origin = bare repo on NAS)
├── herdctl.yaml                     # generated by paddock (fleet defaults; agents added at runtime)
├── <slug>/
│   ├── project.yaml                 # metadata (incl. model)
│   ├── OVERVIEW.md                  # sweeper-curated (auto-committed)
│   ├── CHANGELOG.md                 # sweeper + hand-edited
│   ├── <authored files…>           # notes.md, spec.html, … (you commit these)
│   └── .chats/
│       ├── <session-id>.jsonl       # the chat transcripts (relocated here)
│       └── index.json               # paddock-owned per-project session index (see §6)
└── .gitignore                       # ignores transient caches; tracks content + .chats
```

A `.git` lives **inside** this dir (one repo, never a split git-dir mount). The whole tree is what gets backed up to the NAS bare repo.

## 5. Transcript relocation (SPIKE)

Today the native CLI writes to `~/.claude/projects/<encode(cwd)>/`. To land them in `<project>/.chats/` instead:

- **Recommended: paddock-managed symlink.** On project register/startup, paddock ensures `~/.claude/projects/<encode(projectDir)>` is a **symlink → `<projectDir>/.chats/`**. The CLI writes to the encoded path (the symlink); files physically land in the project dir. herdctl's discovery scans the encoded path and follows the symlink transparently — **no herdctl change needed for the native runtime**. Portability: on a new host/path, paddock recreates the symlink for the new encoded path; the files travelled with the repo. (We already fixed `encodePathForCli` to match Claude Code's encoder, so the symlink name is correct.)
- **Alternative:** a first-class herdctl per-agent/per-trigger *transcript-dir override* (conceptually cleaner, and it would **also unblock Docker isolation #7** — same root cause). Under the hood it would likely still use a symlink or `CLAUDE_CONFIG_DIR`. Defer; the symlink gets us there now.
- **SPIKE to confirm:** the CLI follows the symlink for both write and `--resume`; behaviour when `.chats/` doesn't exist yet (paddock pre-creates it); and that discovery + delete + rename still resolve through the symlink.

## 6. Attribution portability (SPIKE)

herdctl lists a project's chats by filtering on `.herdctl/jobs` attribution (we hit this migrating the water-heater chat — had to synthesize a job record). For self-contained projects, attribution must travel with the project.

- **Recommended:** paddock owns a per-project **`.chats/index.json`** (sessionId → {createdAt, model, name, origin}). paddock discovers a project's chats by listing `.chats/*.jsonl` + this index, rather than relying on herdctl's global attribution. This decouples portability from herdctl state **and** makes migration trivial (drop the JSONL + add an index entry — no job-yaml synthesis).
- Transitional: until paddock owns discovery, persist `.herdctl/` to the NAS too so attribution isn't lost on LXC rebuild.
- **SPIKE:** decide how much of paddock's chat listing moves off `getAgentSessions` onto the local index (resume still goes through herdctl with an explicit `resume`, which already works).

## 7. Sync to NAS + git auth

- **NAS = a bare repo** (`git init --bare` on an NFS export / SSH path). paddock's git layer commits locally and `git push origin` on a cadence (after sweeps/turns, or every N minutes if dirty). LXC rebuild = clone.
- **Auth, tiered:**
  - **SSH deploy key** — universal baseline (GitHub, GitLab, Gitea, the NAS bare repo). Generated on the LXC; public half added as a write deploy key / NAS authorized_key. paddock just `git push`.
  - **GitHub OAuth device flow** — the in-app "Connect GitHub" UX: paddock shows a code, you approve at github.com/login/device, paddock stores a scoped token. No callback URL needed.
  - **GitHub App** — productized/multi-user path (install per repo, short-lived tokens). Later.
- paddock commits under a **configurable identity** (default `Paddock <paddock@…>`).

## 8. Commit model (default; tunable)

- **Auto (for durability):** OVERVIEW.md, CHANGELOG.md, `.chats/**` are auto-committed + pushed by paddock with machine messages (`chat: <slug> turn`, `sweep: <slug>`), so nothing is ever unbacked-up.
- **Manual (for curation):** authored files surface in a per-project **"Changes"** view (git status + diff) with a **Commit + Push** button and a message field. This is the Zellij-pile-up fix: you always see what you've added-but-not-checkpointed.
- Open: whether to also auto-snapshot authored changes to a safety ref between manual commits (belt-and-suspenders durability). Lean: yes, a low-noise `paddock/autosave` ref, invisible in the main history.

## 9. Component breakdown

- **herdctl:** no change required for the native symlink approach. (Optional later: first-class transcript-dir override → also fixes Docker #7.)
- **paddock server:** a `GitBackingStore` capability over `projectsRoot` (detect repo; per-project `status`/`diff`; `commit`; `push`; device-flow auth + token store); symlink management for `.chats/`; per-project `.chats/index.json`; auto-commit hooks on sweep/turn; REST: `GET …/git/status`, `GET …/git/diff`, `POST …/git/commit`, `POST …/git/push`, `GET/POST /api/git/connect` (device flow), `GET /api/git/remote`.
- **paddock web:** per-project "Changes" indicator + panel (status, diff, commit message, Commit + Push); a "Connect GitHub" settings flow (device-flow code + status); a sync/last-pushed indicator.
- **deploy:** LXC mount/remote config (local working tree, NAS bare repo as origin, deploy key); doc the rebuild-from-clone runbook.

## 10. Phased build plan

- **Phase 1 — git-aware read surface (safe, no data changes).** Detect repo; `GET /api/projects/:slug/git/status` (porcelain → structured); web "N uncommitted changes" indicator + a read-only Changes/diff panel. No-ops when `projectsRoot` isn't a repo. *Ships value immediately; risk-free.*
- **Phase 2 — commit + push + auth.** Device-flow "Connect GitHub"; commit/push endpoints + UI; configurable identity; auto-commit generated files.
- **Phase 3 — transcript relocation.** The symlink spike; `.chats/` layout; per-project `.chats/index.json`; migration of existing transcripts into project dirs.
- **Phase 4 — NAS deployment.** Bare repo on NAS; deploy key; continuous push; convert prod `/var/lib/paddock/projects` into the repo; rebuild-from-clone runbook.
- **(Later) herdctl transcript-dir override** → unify native + Docker (#7).

## 11. Risks / open questions

- Git over network FS (if we ever mount the working tree): avoided by Option B, but if mounted, **NFS only, never SMB**.
- Symlink correctness across encode/resume/delete (Phase 3 spike).
- Auto-commit cadence vs history noise (§8).
- Transcripts in git: committed as **regular files** for now (JSONL is text, compresses fine; no LFS infra). Revisit LFS only if repo size becomes a problem.
- Secrets: the device-flow token + deploy key live on the LXC (env / restricted file), never committed, never on the NAS share.
